# Blobby — Written Reflection

**What does Blobby do?**
Blobby is an AI-powered summarization tool that accepts multiple input types — PDFs, images with text, audio files, and pasted text — and returns a structured summary generated by a large language model. The goal is to reduce the friction of digesting large or varied content formats by centralizing them into a single interface. A user uploads or pastes their content, and Blobby returns a readable, organized summary without requiring them to switch between tools.

**What architecture did you choose and why?**
Blobby uses a tiered Bronze/Silver/Gold architecture that separates input handling, processing, and output rendering into distinct layers. This structure was chosen because it keeps each concern isolated; if the summarization prompt changes, it only affects the processing layer without touching input parsing or the UI. It also makes the system easier to extend — adding a new input type means adding a handler at the Bronze layer without rewriting core logic. I chose this over a flat single-function approach because the variety of input types made a layered structure the more maintainable option.

**What alternative did you consider and why did you reject it?**
I considered integrating a local LLM to avoid API dependency and reduce cost, but rejected it because local models at the time of development couldn't match the summarization quality of hosted models for diverse input types, especially audio and low-quality images. Running a local model also introduces hardware requirements that make the app inaccessible to users without capable machines. Using a hosted API kept the app lightweight and deployable without setup friction.

**What capability did you NOT implement and when would you add it?**
I did not implement video summarization. Blobby currently handles static and audio content but not video files. I would add this in a future iteration using a transcription step (e.g., Whisper) piped into the existing summarization pipeline — the audio handling infrastructure already partially supports this pattern. The main reason it was cut was time; video processing also adds significant latency and cost that would need UX handling I hadn't built yet.

**What did your evaluation show?**
Running five test cases across the supported input types showed that Blobby performs well on clean, text-heavy PDFs and pasted content, producing structured and accurate summaries. Performance degraded on low-quality or ambiguous images, where the model either hallucinated content or returned vague output. Audio summarization worked but was sensitive to recording quality. Two failure cases — a blank image and an invalid API key — confirmed that the app lacks graceful error handling in edge cases; the blank image produced a confabulated summary rather than flagging that no content was detected.

**What did you change based on evidence and what improved?**
Based on evaluation, I rewrote the prompt in `lib/summarize.ts` to be more structured and explicit. The original prompt was a generic instruction with no specified format or failure handling. The updated prompt assigns the model a clear role, specifies output format (sections with headers), and instructs the model to flag low-confidence inputs rather than guessing. After the prompt change, output on ambiguous inputs improved noticeably — the model was more likely to acknowledge uncertainty than fabricate structure, which is the more honest and useful behaviour for a summarization tool.
